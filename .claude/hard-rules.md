# Hard Rules & Per-PR Checklist

`AGENTS.md` lists the rules as one-liners. This file gives the rationale,
concrete examples, and the per-PR checklist.

## Rules with rationale

These are the failure modes the project has hit before and you must not
repeat.

1. **Don't edit only one Spark version.** If a file exists in multiple
   `spark-<v>/` trees, decide upfront which versions you'll touch. The CI
   matrix catches mismatches and the human reviewer is annoyed by them. See
   [`code-layout.md`](code-layout.md#cross-version-duplication-is-intentional).

2. **Don't use Spark API symbols that don't exist in all targeted versions.**
   Check the Scaladoc for the lowest Spark version you're modifying.
   See [`code-layout.md`](code-layout.md#cross-version-drift).

3. **Don't introduce new Gradle modules or change `settings.gradle`'s
   parameterization scheme** without an explicit design discussion. The
   per-Spark-version layout is load-bearing for the release process.

4. **Don't write to the local `docs/` directory by hand.** It's deprecated
   (see [Documentation policy](#documentation-policy) below). The single
   exception is `docs/configurations/*.md`, regenerated by `ConfigurationSuite`
   (see [`build-and-test.md`](build-and-test.md)).

5. **Don't bypass the Java client.** All ClickHouse I/O goes through
   `com.clickhouse:client-v2` via `NodeClient` / `ClusterClient`. No JDBC, no
   raw HTTP, no `com.clickhouse.client.*` V1 imports. The *Java client (V2)*
   section of [`code-layout.md`](code-layout.md) lists the V1 symbols to grep
   against.

6. **Don't silence exceptions on the I/O path.** `NodeClient` and the
   readers/writers must propagate failures. If a retryable error code needs
   handling, wire it through `spark.clickhouse.write.retryableErrorCodes`
   rather than special-casing in code.

7. **Don't leak `AutoCloseable` resources.** `NodeClient` and `ClusterClient`
   are reused via internal caches and the catalog owns the top-level
   instance, but anything you instantiate yourself must be closed on every
   exit path including exceptional ones. Use `Utils.tryWithResource`
   (`clickhouse-core/.../Utils.scala`) or `try`/`finally`.

8. **Don't rename the public-API surface** — the classes, `clickhouse` short
   name, and `com.clickhouse.spark` `groupId` / package listed under *What this
   is* in `AGENTS.md`, plus the `clickhouse` registration in
   `META-INF/services/org.apache.spark.sql.sources.DataSourceRegister`. Users
   depend on all of them.

9. **Every user-visible change needs a `CHANGELOG.md` entry** under
   `[Unreleased]` (`Added` / `Changed` / `Fixed` / `Removed`) with a PR
   link, matching the existing one-line style. *User-visible* = changes
   behavior, output, public API, configs, or the dependency surface that
   users can observe. Pure internal refactors, test-only changes, CI/build
   adjustments, and docs/style fixes don't qualify. Breaking changes
   (default flips, removed configs, renamed public classes, changed type
   mappings) also go under a **Breaking Changes** sub-section and must be
   called out in the PR description with downstream impact considered.
   Template: the 0.10.0 flip of `ignoreUnsupportedTransform` to `true`,
   which required a new `…convertLocal.allowUnsupportedSharding` guard to
   prevent silent distributed-write corruption.

10. **No silent fallbacks.** If a push-down isn't supported for some types,
    return `false` from the appropriate `pushXxx(...)` and log; don't
    produce SQL that ClickHouse rejects only at runtime, and don't let the
    scan continue against a different shape than what Spark expects.

11. **Pushable scalar functions in `func/` must match ClickHouse semantics
    exactly.** These produce SQL that runs server-side; wrong null
    handling, integer width, or hash seed silently corrupts shard routing.

## Workflow checklist (per PR)

Final pre-commit pass. If you can't tick a box, don't open the PR.

- [ ] Each modified file in `spark-<v>/clickhouse-spark/` has an equivalent
      change in every applicable peer module (3.3 / 3.4 / 3.5 / 4.0), or the
      PR description explicitly justifies the exclusion.
- [ ] `./gradlew clean build -x test -Dspark_binary_version=<v>` passes for
      every `<v>` you touched.
- [ ] `./gradlew test ...` passes locally for each touched Spark version,
      with Docker running. (CI also runs the full ClickHouse-version matrix.)
- [ ] `./gradlew spotlessApply -Dspark_binary_version=<v>` has been run for
      each `<v>`, and `spotlessCheck` is clean.
- [ ] Apache-2.0 license header is present on every new source file.
- [ ] If you added/changed an entry in `ClickHouseSQLConf`, you regenerated
      `docs/configurations/*.md` via `UPDATE=1 ConfigurationSuite` for every
      Spark version that exposes the entry (3.4 / 3.5 / 4.0; 3.3 has no
      `ConfigurationSuite`).
- [ ] If the change is user-visible, `CHANGELOG.md` has an entry under
      `[Unreleased]` in the correct section
      (`Added` / `Changed` / `Fixed` / `Removed`) with a PR link.
- [ ] Breaking changes (default flips, removed configs, renamed classes,
      changed type mappings) are called out explicitly in the PR
      description **and** in `CHANGELOG.md` under a **Breaking Changes**
      sub-section.
- [ ] Tests cover the happy path **and** the non-trivial edge cases the
      change actually exposes (nulls, empty partition, type boundary,
      distributed write, etc.). New push-down code has a test that verifies
      the generated SQL.
- [ ] Commit message follows [Conventional Commits](https://www.conventionalcommits.org/)
      (`feat(write): ...`, `fix(read): ...`, etc.) per `CONTRIBUTING.md`.
- [ ] If user-facing docs need updating, the change is sent to
      [`ClickHouse/clickhouse-docs`](https://github.com/ClickHouse/clickhouse-docs),
      not the local `docs/`.

## Documentation policy

- Authoritative user docs:
  <https://clickhouse.com/docs/en/integrations/apache-spark>, sourced from
  the [`ClickHouse/clickhouse-docs`](https://github.com/ClickHouse/clickhouse-docs)
  repo. Submit doc PRs there.
- The local `docs/` directory is **deprecated** and slated for removal.
  Don't extend it.
- Inline Scaladoc / Javadoc on public API (`ClickHouseCatalog`,
  `ClickHouseTableProvider`, `ClickHouseTable`, public methods of
  `NodeClient`) must match the implementation. Doc lies are real bugs.
- Update `README.md` only when the user-facing surface (supported version
  matrix, basic usage snippet) changes.
